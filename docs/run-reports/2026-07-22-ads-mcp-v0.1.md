# ads-mcp v0.1 local implementation

## Outcome

Implemented the approved local stdio MCP boundary in `packages/ads-mcp` without adding remote
hosting, OAuth, a database, an agent graph, autonomous revision loops, or implicit model calls.

The server exposes exactly three tools:

1. `ads_render` captures an allowed URL or root-confined TSX component with the existing ADS
   Chromium evidence engine.
2. `ads_evaluate` normalizes deterministic gates, optionally runs the existing screenshot
   comparison engine, and returns `needs_human` when judgment is unresolved.
3. `ads_trace` verifies final decisions against render-time file hashes, exact rule and source
   excerpts, implementation artifacts, and resolvable evidence from the same run.

Run artifacts are filesystem-backed under `<root>/.ads/runs`. Initial runs and repeated evaluation
or trace stages use temporary directories plus atomic renames. Latest-stage indexes point to
immutable stage receipts instead of overwriting prior evidence.

## Contract adjustments discovered during implementation

- Added explicit `provenance.artifactFiles`. A URL render cannot know which implementation files
  later decisions will cite, so artifact hashes must be captured prospectively.
- Added `platform`, `renderer`, and `detectors` to the run manifest. The public three-tool protocol
  remains stable while later platform adapters can identify their evidence source.
- Logged SwiftUI as the first candidate non-web adapter, using `impeccable-swift` as a reference
  rather than a dependency. This remains outside v0.1.

## Security boundary

- Project reads and run writes are confined to the configured root with traversal and symlink
  checks.
- URL targets require HTTP(S). Exact localhost hosts are allowed by default; other origins require
  explicit startup allow-listing.
- Embedded URL credentials are rejected. Common secret query parameters are redacted from stored
  receipts.
- Components are bundled without executing project code in Node, then served from an ephemeral
  `127.0.0.1` port for browser capture.
- MCP resources expose only allow-listed run artifacts and screenshots recorded in evidence.
- Tool cancellation and configured timeouts terminate the browser or comparison subprocess.

## Verification

- `npm run mcp:test`: 12/12 tests passed.
- Real Chromium URL capture passed authoritative ADS gates.
- Real TSX component capture passed authoritative ADS gates.
- Screenshot comparison passed through the packaged existing ADS comparison engine.
- The compiled stdio binary completed `ads_render -> ads_evaluate -> ads_trace` through a real MCP
  client, including a resource read.
- MCP Inspector `tools/list` passed and exposed exactly the three contracted tools with input and
  output schemas.
- Package dry-run contains the binary, declarations, source maps, and vendored ADS capture and
  comparison engines; test files are excluded.
- Root `npm run release:check` passed after installing the clean checkout's existing root
  dependencies. All ADS release and frozen regression gates remained green.
- `git diff --check` passed.

## Failed checks and open risks

- The first root release-gate attempt stopped because the worktree had no root `node_modules` and
  `pngjs` could not resolve. `npm ci` restored the locked dependencies; the full release gate then
  passed. This was an environment prerequisite, not a source failure.
- `npm audit` reports two moderate advisories through `@modelcontextprotocol/sdk@1.29.0` and its
  `@hono/node-server@1.19.14` dependency. The affected Hono static-serving path is not used by this
  stdio-only server. Recheck before adding remote HTTP or publishing; do not force a major transitive
  override without SDK compatibility evidence.
- v0.1 implements `judge.mode: none` only. A passing deterministic run correctly remains
  `needs_human` until a visual judge is explicitly configured in a later version.
